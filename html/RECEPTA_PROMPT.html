RECEPTA · Documentation de conception & Prompts IA

01 Vue d'ensemble du projet

RECEPTA est une application web SaaS Flask (Python) destinée aux ingénieurs et techniciens BTP. Elle automatise la réception topographique de chantiers routiers : comparaison côtes mesurées / côtes théoriques, génération de fiches PDF officielles, archivage cloud, workflow MDC ↔ Entreprise de Travaux.

Produits
RECEPTA SOLO — accès direct, un seul utilisateur par projet
RECEPTA PRO — workflow MDC/ET, demandes de réception, validation, partenariat
Modes de réception
Assainissement (ASS) — 9 colonnes, profil en travers drainage
Terrassement (TER) — 11 colonnes, dist. axe + pente
Rôles utilisateurs
admin — gestion globale plateforme
MDC — Mission de Contrôle, valide les réceptions
ET — Entreprise de Travaux, soumet les demandes
Données Excel V3
Onglets : GEN AXE TER ASG ASD ASS GEO DIV PK_Coordonnees OFFSETS_TRANSVERSAUX

PROMPT MASTER · RECEPTA
Tu travailles sur RECEPTA, une application web SaaS Flask (Python) pour la réception topographique BTP. == IDENTITÉ == Nom commercial : RECEPTA (en cyan #00ffff, lettres capitales, espacement 2px) Suite : OPTILAB (sous-titre "by OPTILAB", discret) Logo : grille CSS 2×2 points (#00ffff / #fbbf24 / #00ffff / #00ffff) — jamais d'image Fond principal : #07101c (navy très sombre) Accent principal : #00ffff (cyan) Accent secondaire : #fbbf24 (gold/ambre) MDC = violet #7c3aed | ET = ambre #f59e0b == CONTEXTE MÉTIER == Secteur : BTP, chantiers routiers/autoroutiers, assainissement, terrassement Utilisateurs : Contrôleurs topographiques (MDC), Techniciens Entreprise de Travaux (ET), Admin Objectif : Comparer les côtes mesurées sur le terrain aux côtes théoriques issues du modèle Excel, calculer les écarts, valider la conformité selon une tolérance (en cm), générer une fiche de réception PDF officielle avec signatures manuscrites, l'envoyer par email, l'archiver. == MODES DE RÉCEPTION == ASS (Assainissement) : 9 colonnes — PK, Élément, LAV, Côte LAV, Mesurée, Théorique, Écart+, Écart−, Obs TER (Terrassement) : 11 colonnes — idem + Dist. axe, Pente == WORKFLOW EN 4 ÉTAPES == 1. Informations générales — Projet, Date, Opérateur, Section, Météo, Tolérance, Mode (ASS/TER) 2. Sélection — cocher les PK et les éléments (gauche/axe/droite) à mesurer 3. Saisie mesures — tableau par station (Cote repère, LAV, puis lignes PK×Élément) - Écart = Côte mesurée − Côte théorique (signe conservé) - Statut : ecart-ok (|Δ| ≤ tol), ecart-positif (Δ > 0 hors tol), ecart-negatif (Δ < 0 hors tol) - row-interpolated : point interpolé entre deux PK, exclu du bilan - Validation par ligne : — / ✓ / ⚠ (cyclable) - Vue Tableau ou Vue Cartes (mobile-friendly) 4. Signatures — MDC + ET (canvas tactile), Observations générales, Verdict (Validée/Non validée/À reprendre) == RÉSULTATS == KPIs : total points, % conformité, nb conformes, nb hors tolérance Tableau coloré, PDF généré, envoi email Resend, archivage R2 == MODÈLE EXCEL V3 == Onglets : GEN, AXE, TER, ASG (côté gauche), ASD (côté droit), ASS, GEO, DIV, PK_Coordonnees, OFFSETS_TRANSVERSAUX Parsing : pandas + openpyxl, NaN/Inf → null, colonne PK cherchée par nom 'PK' ou 'KILOMETRIQUE' Stockage : Cloudflare R2 via boto3 (clé : data/clients/{client_id}/modele.xlsx) == RÔLES & ACCÈS == admin → /admin/* — gestion globale MDC → role = 'mdc' sur MembreProjet — initie réceptions, valide demandes ET ET → role = 'entreprise' sur MembreProjet — soumet demandes Plan SOLO → 1 utilisateur/projet, pas de workflow demandes Plan PRO → MDC + ET distincts, demandes de réception, validation, rapport période == STACK == Backend : Flask + SQLAlchemy + PostgreSQL (Supabase) Auth : Flask-Login (@login_required / @pro_required / @admin_required) PDF : xhtml2pdf (WeasyPrint retiré — pas de GTK sur Render) Email : Resend API, pièce jointe base64 Cloud : Cloudflare R2 (boto3) Frontend : Vanilla JS, sessionStorage inter-pages, Tailwind CDN (admin/pro), Chart.js, Leaflet+proj4 Déploiement : Render (gunicorn wsgi:app), Python 3.11 == DESIGN RULES == - Fond dark navy #07101c, cards #0d1b2e/#0f2038, border rgba(0,255,255,.08) - Boutons primaires : background cyan → color dark navy - Police : Inter (app), Bebas Neue (landing/grands titres) - Pas de mode clair dans l'app (seulement option toggle sur landing) - Pas d'emojis sauf validation cyclable (✓ ⚠) et indicateurs - Commentaires code en français - Vanilla JS uniquement — pas de React/Vue/jQuery - SVG inline pour les icônes (pas de Font Awesome) == FICHIERS PRINCIPAUX == app/blueprints/pages.py → routes HTML principales app/blueprints/api.py → API REST (/api/*) app/blueprints/auth.py → /login /logout app/blueprints/admin.py → /admin/* app/blueprints/pro/projets.py → /pro/projets/* (hub, rapport, visualisation) app/blueprints/pro/demandes.py→ /pro/demandes/* (soumission, historique) app/models.py → Projet, User, Client, MembreProjet, ClientProjet, DemandeReception, FicheReception app/services/excel_service.py → parse_modele_config(), parse_profil_long() app/services/pdf_service.py → make_pdf_bytes_any() app/services/r2_service.py → upload/download/presigned_url app/services/visualization_service.py → generate_section_svg() app/templates/reception_topographique.html → 4000+ lignes, page centrale app/templates/pro/projet_hub.html → hub projet PRO app/templates/admin/projet_visualisation.html → 7 onglets admin == CONVENTIONS == - Toujours lire CLAUDE.md avant de modifier le code - Ne jamais écraser des valeurs déjà saisies (auto-fill seulement si champ vide) - Gestion d'erreur API : {"error": str(e)}, 500 - NaN/Inf dans pandas → None avant JSON - Pièce jointe email : base64.b64encode(pdf_bytes).decode()
10 Prompt UI/UX — Interface de réception
PROMPT · RECEPTION UI
Conçois l'interface de réception topographique RECEPTA en respectant ces règles : CONTEXTE : L'utilisateur est un topographe sur un chantier BTP, souvent en plein soleil sur une tablette ou un smartphone. L'interface doit être lisible en plein air, avec des cibles tactiles généreuses (min 44px), et un workflow sans ambiguïté. ÉTAPE 3 — SAISIE DES MESURES (la plus critique) : Structure d'un tableau par station : - En-tête station : numéro, plage PK, [Cote repère] [LAV] [Pente %] - Colonnes (ASS 9 cols) : PK | Élément | LAV calc | Côte LAV | [Mesurée] | Théorique | Δ+ | Δ− | ✓/⚠/— | 📷 - Colonnes (TER 11 cols) : + Dist. axe | Pente - Chaque ligne porte une classe CSS : ecart-ok / ecart-positif / ecart-negatif / row-interpolated - Couleurs : vert (#22c55e) = conforme, rouge (#ef4444) = hors tol, gris (#94a3b8) = interpolé - Les lignes gauche (G_*) = teinte rouge-pâle, droite (D_*) = teinte bleue-pâle - Bouton "+" entre deux PK = insérer un point interpolé entre ces deux PK - Vue Tableau vs Vue Cartes : toggle dans le header du step (pas un tweak panel) VUE CARTES (mobile) : - Une card par combinaison PK×Élément - Affiche : PK en grand, Élément, [input Mesurée], Théo, Δ, statut dot coloré - Disposition : grille 2 colonnes sur tablette, 1 colonne sur mobile VALIDATION CYCLABLE : - Bouton dans colonne "Valid." : — → ✓ → ⚠ → — (cycle au clic) - — = en attente (gris), ✓ = validé (vert), ⚠ = à reprendre (orange) INTERPOLATION : - Petit bouton "+" discret entre deux lignes de PK différents - Au clic : modal/inline pour saisir PK1 et PK2 (pré-remplis), confirmer - La ligne interpolée a class row-interpolated, est grisée, exclue du bilan OBSERVATIONS PAR LIGNE : - Icône 📷 → ouvre un mini-panneau : titre libre + commentaire + photo (caméra ou fichier) - Ne pas bloquer le workflow principal STATUT CONFORMITÉ : - Lire row.classList.contains('ecart-positif') ou 'ecart-negatif' pour le bilan - ecartNeg contient déjà le signe −, ne pas re-préfixer - Bilan : count des lignes, exclure row-interpolated
